Two kinds of evidence: a career building reliable data systems for regulated institutions, and a personal AI system I run on the same architecture every day.

Flagship project

de Volksbank & ASN — where Model-Driven Data Engineering was built.

As Lead Modeller, I led the design of a new, model-driven data warehouse for de Volksbank's brands (ASN Bank, SNS, RegioBank, BLG Wonen). It became one of the most influential MDDE implementations in the Dutch data community — and the proving ground for the method I now bring to other teams.

The idea

Separate what the data means (functional modeling — business semantics, relationships) from how it runs (technical engineering). Merge structure (the model) and movement (the transformations) into one unified metadata framework.

How it works

Rules and models are defined as structured metadata (entity-first), not hand-coded pipelines. The platform reads those models and generates database-agnostic transformations, data-quality tests and documentation — consistently, across the bank.

Why it mattered

Business stakeholders could understand and trust the data because physical designs are regenerated from logical models. Lineage and historization are built in. Manual pipeline work became architecture-as-code.

The payoff

Agility and independence from any single database technology, far fewer hand-coding errors, and governance that's inherent rather than bolted on. Delivered with continuous user involvement and short cycles — not waterfall.

I invited CrossBreeze (Harmen Wessels & Willem Otten), whom I'd met on the Rabobank project, to help build the framework — and brought the wider community along through the MDDE meetup and Roelant Vos's training. The trade press covered it (see below). This is the work the rest of my approach is built on.

Signature engagement

Migrating legacy SQL — and proving it's identical.

My largest recent delivery: migrating complex legacy SQL to a governed, bi-temporal integration layer for a bank's Risk & Finance data platform. The hard part isn't rewriting the SQL — it's proving the new pipeline produces exactly the same data as the old one. So I built the tooling to prove it.

CTE-by-CTE regression testing

Comparing only the final output tells you that something differs across 50,000 rows. Comparing query step by step tells you which transformation introduced it — root cause found, not just symptom. Migration debugging, ~10× faster.

The CTE Notebook generator

Explode a SQL file into an interactive notebook where each step is profiled and checked — PII detection, SCD2/temporal patterns, lineage, data contracts — before a single query runs. SQL becomes self-documenting and self-validating, then exports back to deployable SQL.

Determinism before AI

Parse legacy SQL into metadata, deterministically rebuild clean SQL that fits the target architecture, prove equivalence with automated regression — then let AI safely optimize the rest. Non-deterministic SQL (ROW_NUMBER without a stable ORDER BY) caught before it corrupts the validation.

This is where the method earned its stripes: systematic validation and large-scale SQL analysis turned migration from a risky rewrite into something verifiable. The same discipline now underpins how I make AI output reliable. More in the writing →
