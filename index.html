The stance

Most people experience AI as a magic box.

Type something clever, something impressive comes out, and tomorrow it remembers nothing. Impressive demo, unreliable in real work.

The architect's stance is the opposite. Output quality is a function of the structure around the model — the context it can see, the memory it carries, the rules that encode your standards, the data wired in. Get the structure right and good output stops being luck. It becomes repeatable. That's the difference between a toy and an instrument.

It's where the whole field is heading. More and more code is generated by AI — and it only produces work you can trust if it has a solid foundation to generate against: clear rules, curated patterns, a model of what "correct" means. Building that foundation is exactly what I've done for 25 years.

One concrete example

A risky SQL migration, made verifiable.

Migrating transformations between platforms usually means a rewrite you cross your fingers on. Comparing only the final output tells you that something differs across 50,000 rows — not which step caused it.

I built CTE-by-CTE regression testing: compare each transformation step before and after, so the diverging step is pinpointed, not guessed. Root cause found, not just symptom — migration debugging roughly 10× faster. The same discipline now underpins how I make AI output reliable.
